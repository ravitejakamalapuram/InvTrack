# Play Store Changelog Guide

## Overview

This guide explains how to create effective Play Store changelogs that are user-friendly and comply with the 500-character limit.

Play Store changelogs are **automatically generated** by the CI/CD workflow when code is merged to `main`.

---

## Automated Generation (CI/CD)

### How It Works

When you merge to `main`, the **CD: Version & Changelog** workflow automatically:

1. Determines version bump (major/minor/patch) from conventional commits
2. Generates user-friendly changelog using `cliff-playstore.toml`
3. Creates `android/fastlane/metadata/android/en-US/changelogs/{version_code}.txt`
4. Validates character count (≤500)
5. Commits the changelog file
6. Creates a git tag for release

**No manual intervention needed!** Just write good commit messages.

### Manual Generation (If Needed)

```bash
# Generate Play Store changelog manually
git-cliff --config cliff-playstore.toml --latest --strip all

# Save to specific version code
git-cliff --config cliff-playstore.toml --latest --strip all > \
  android/fastlane/metadata/android/en-US/changelogs/89.txt
```

## Writing User-Friendly Commit Messages

### ✅ Good Commit Messages (User-Facing)

```
feat: View PDFs directly in the app
fix: Resolved crash when opening large documents
perf: Faster portfolio loading on startup
feat: Added dark mode support for all screens
```

### ❌ Bad Commit Messages (Too Technical)

```
feat: Implement DocumentViewerScreen with pdf_viewer package
fix: NullPointerException in InvestmentRepository.getAll()
refactor: Extract FireCalculationService to separate file
```

### Best Practices

1. **Start with user benefit**: What does the user get?
2. **Be specific**: "View PDFs in-app" not "Add document support"
3. **Avoid jargon**: "Faster loading" not "Optimized async operations"
4. **Skip internal details**: Users don't care about refactoring

## Changelog Configurations

### `cliff.toml` (Full Changelog)

- Used for: GitHub releases, CHANGELOG.md
- Format: Detailed with all commit types
- No character limit
- Includes technical details

### `cliff-playstore.toml` (Play Store)

- Used for: Play Store release notes
- Format: Compact, user-friendly
- Limit: 500 characters
- Only shows: Features, Bug Fixes, Performance, Security
- Skips: Documentation, Refactoring, Tests, CI/CD

---

## Manual Editing Tips

### Character Budget

- **Total limit**: 500 characters
- **Recommended**: 400-450 (leaves room for edits)
- **Minimum**: Include at least 1-2 key features

### Structure

```
🆕 What's New
• [Most important feature]
• [Second feature]

🐛 Bug Fixes & Improvements
• [Critical fix]
• Performance and stability improvements
```

### Writing Tips

1. **Lead with value**: "Track your FIRE progress" not "Added FIRE feature"
2. **Use active voice**: "View PDFs" not "PDFs can be viewed"
3. **Be concise**: "Faster loading" not "Significantly improved loading performance"
4. **Group minor fixes**: "Bug fixes and improvements" instead of listing each

## Examples

### ❌ Bad (Too Technical, 153 chars)

```
## [3.32.0] - 2026-02-04

### ✨ Features

- Add in-app Help & FAQ screen with usage information

---
*Generated by [git-cliff](https://git-cliff.org)*
```

**Issues:**
- Version header wastes space (Play Store shows version)
- Footer wastes 50 characters
- Commit message is technical, not user-facing

### ✅ Good (User-Friendly, 147 chars)

```
🆕 What's New
• Get instant answers with the new in-app Help & FAQ
• Learn how to track investments and understand XIRR

🐛 Bug fixes and stability improvements
```

**Why it's better:**
- No wasted space on headers/footers
- Explains user benefit ("Get instant answers")
- Adds context ("Learn how to...")
- Groups minor fixes

## Workflow Integration

### During Development

1. Write user-facing commit messages
2. Use conventional commits: `feat:`, `fix:`, `perf:`
3. Think: "How would I explain this to my mom?"

### On Merge to Main

The **CD: Version & Changelog** workflow (`.github/workflows/cd-version.yml`) automatically:

1. **Determines version bump** from commit messages:
   - `BREAKING CHANGE` or `!:` → major version (3.0.0)
   - `feat:` → minor version (3.1.0)
   - `fix:`, `perf:`, etc. → patch version (3.0.1)

2. **Generates changelog** using `cliff-playstore.toml`:
   - User-friendly format (no technical jargon)
   - Removes scope prefixes (e.g., "feat(fire):" → just the message)
   - Only includes user-facing changes
   - Validates ≤500 character limit

3. **Creates files**:
   - `android/fastlane/metadata/android/en-US/changelogs/{version_code}.txt`
   - Updates `CHANGELOG.md` (full changelog with `cliff.toml`)
   - Updates `pubspec.yaml` version
   - Updates `about_screen.dart` version

4. **Commits and tags**:
   - Commits all version files with `chore(release): vX.Y.Z [skip-release]`
   - Creates git tag `vX.Y.Z`

### On Tag Push

The **CD: Deploy to Play Store** workflow (`.github/workflows/cd-deploy-android.yml`) automatically:

1. Builds release bundle
2. Uses the generated changelog from `changelogs/{version_code}.txt`
3. Deploys to Play Store alpha track
4. Creates GitHub Release with changelog

## Common Patterns

### Feature Release

```
🆕 What's New
• [Main feature with user benefit]
• [Secondary feature]
• [Tertiary feature]

🐛 Improvements
• Bug fixes and performance enhancements
```

### Bug Fix Release

```
🐛 Bug Fixes
• Fixed [specific user-facing issue]
• Resolved [another issue]
• Performance and stability improvements
```

### Mixed Release

```
🆕 New Features
• [Key feature]

🐛 Fixes & Improvements
• Fixed [critical bug]
• Faster [specific operation]
• General stability improvements
```

## Troubleshooting

### Changelog Too Long (>500 chars)

1. Remove version header (saves ~30 chars)
2. Remove footer (saves ~50 chars)
3. Combine minor fixes: "Bug fixes and improvements"
4. Shorten descriptions: "View PDFs" not "View PDF documents directly"
5. Limit to 2-3 most important changes

### Changelog Too Short (<100 chars)

1. Add context: "Track your FIRE progress with visual milestones"
2. Explain benefits: "Know exactly when you'll achieve financial independence"
3. Add "Performance and stability improvements" if needed

### No User-Facing Changes

```
🐛 Improvements
• Performance optimizations
• Bug fixes and stability improvements
• Under-the-hood enhancements for better reliability
```
