## 2024-05-24 - [Secure Storage] Enable encryptedSharedPreferences for FlutterSecureStorage on Android
**Vulnerability:** The `FlutterSecureStorage` plugin was being used on Android with default options (`const AndroidOptions()`). In older configurations, this defaults to using standard, unencrypted `SharedPreferences` for storing the encrypted payload, with only the key stored in the KeyStore.
**Learning:** For modern Android security standards, it is crucial to explicitly enable `encryptedSharedPreferences: true` when configuring `FlutterSecureStorage`. This ensures that both the keys and the values are encrypted using Android's native `EncryptedSharedPreferences`, providing an additional layer of defense against local extraction.
**Prevention:** Always verify that `FlutterSecureStorage` configurations explicitly set `encryptedSharedPreferences: true` in `AndroidOptions` for new applications or features. Note: Migrating an existing app requires careful handling to transfer data from the old unencrypted preferences to the new encrypted ones.

## 2025-01-20 - [IDOR] Fix cross-account local data access vulnerability in DocumentStorageService
**Vulnerability:** In `DocumentStorageService`, the `_isSafePath` function validated document paths against the global `getApplicationDocumentsDirectory()`. This allowed any authenticated user to potentially access documents belonging to other users on the same device by manipulating the `localPath` in their synced Firestore documents (Insecure Direct Object Reference / Path Traversal).
**Learning:** When storing sensitive user data locally on shared devices, path validation must always be scoped to the specific user's isolated directory, not the application's root data directory. Validating against the global app directory is insufficient for multi-account systems.
**Prevention:** Always construct and validate file paths using a user-specific identifier (e.g., `userId`) in the path hierarchy, and ensure validation checks (like `isWithin`) use the user-scoped directory (`_documentsDirectory`) rather than the global app directory.

## 2025-02-18 - [Secure Storage] Store temporary exported files in cache directory
**Vulnerability:** The application stored the exported CSV template in `getApplicationDocumentsDirectory()`, which is a persistent directory meant for important user data and may be backed up to the cloud. This pollutes the persistent user documents directory and risks unintentional data retention.
**Learning:** When temporarily storing files intended solely for external sharing (e.g., exporting CSV templates via `share_plus`), use `getTemporaryDirectory()` instead of `getApplicationDocumentsDirectory()`. This prevents polluting the persistent, often cloud-backed, user documents directory and ensures temporary files can be automatically cleaned up by the OS when space is needed.
**Prevention:** Always use `getTemporaryDirectory()` for files that are ephemeral or only created to be shared with other apps. Only use `getApplicationDocumentsDirectory()` for files that the application needs to retain across sessions.
